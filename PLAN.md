# Geo 协作发布平台计划

> 本文件对应用户口径中的 `plan.md`。当前 Windows 工作区曾存在 `PLAN.md`，文件名大小写可能由系统显示决定。

## 目标

从本地 exe MVP 升级到可长期运行的云端 Web 发布系统，优先支持头条号。系统需要支持多账号登录态隔离、发布任务排队、人工介入、失败恢复、飞书通知、MySQL 持久化，并在 2026 年 6 月 7 日前完成第一版上线。

## 关键边界

- 先只做头条号，后续再扩展其他平台。
- 允许用户通过浏览器完成扫码、验证码、重新登录和必要人工操作。
- 每个发布任务独立浏览器/profile，任务结束后关闭浏览器。
- 登录状态必须按平台和账号隔离。
- 目标并发上限先按 5 个发布浏览器控制，任务多于浏览器时排队。
- 自动发布默认不再要求人工确认，但遇到登录失效、验证码、扫码、阻塞弹窗等情况要暂停并通知。
- 弹窗优先脚本关闭，无法关闭时进入人工介入。
- 数据库后续从 SQLite 切换到 MySQL。
- 第一阶段验收以“能稳定发出第一篇文章并可恢复异常”为核心。

## 当前优先级

1. 把云端浏览器发布 Spike 收敛到后端服务，不再依赖手工启动命令。
2. 建立浏览器会话管理：Xvfb、x11vnc、websockify、Chromium 的生命周期由后端托管。
3. 完成 `waiting_user_input` 的前端入口，让用户能进入对应 noVNC 页面处理扫码、验证码和弹窗。
4. 为后续 MySQL 和账号鉴权预留清晰边界，避免继续把本地单用户假设写死。

## 技术实施建议

- 浏览器运行层独立成服务模块，负责分配 display、端口、profile 路径、进程启动和清理。
- 发布任务只关心“拿到一个可操作的浏览器上下文”，不要直接管理 noVNC 细节。
- 人工介入状态统一使用 `waiting_user_input`，恢复后重新进入 pending 队列。
- 同账号发布互斥必须保留，避免一个账号同时打开多个发布页面导致登录态或草稿混乱。
- MySQL 切换前先收敛数据库访问层和迁移脚本，避免业务代码直接依赖 SQLite 特性。
- 飞书通知先做抽象接口和日志落库，再接真实 API。

## 验收口径

- 云服务器可通过 Web/noVNC 进入指定账号浏览器。
- 头条账号登录态可保存、复用、隔离。
- 单篇文章可自动填充标题、正文、封面和正文图片。
- 正常链路可自动发布成功。
- 遇到扫码、验证码、登录失效、阻塞弹窗时任务进入人工介入状态。
- 人工处理后可恢复任务，不需要重建任务。
- 后端能限制并发，超出任务进入队列。
- 失败有日志、截图或可定位信息。
