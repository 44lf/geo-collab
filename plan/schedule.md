# 一个月开发排期

目标：20 个工作日内交付 Windows 本地 MVP。排期按“可验收里程碑”组织，避免只按模块堆任务。

## 总体节奏

| 周期 | 里程碑 | 交付判断 | 任务文件 |
| --- | --- | --- | --- |
| Week 1 | 工程能跑，关键风险有结论 | 后端、前端、数据库骨架可启动；头条号登录和发布页填充 Spike 有结果 | `week-1-foundation-risk.md` |
| Week 2 | 内容和账号可用 | 文章/资源/分组闭环；头条号账号授权、校验、重登、导出入口可用 | `week-2-content-account.md` |
| Week 3 | 发布主链路打通 | 能创建任务、轮询分配、自动填充头条号发布页、等待人工确认、记录结果和重试 | `week-3-task-publish.md` |
| Week 4 | 可打包可试用 | exe 可启动应用；Playwright 可用；主链路验收通过；交付说明完成 | `week-4-packaging-acceptance.md` |

## Week 1：骨架与风险验证

目标不是完成所有基础设施，而是尽早回答两个问题：

- 应用是否能按 FastAPI + React + SQLite 的形态跑起来。
- 头条号登录状态复用、发布页填充和封面上传是否可行。

本周不可延期：

- `W1-01` 项目结构和启动脚本。
- `W1-02` SQLite/Alembic 最小迁移。
- `W1-04` 头条号登录状态保存 Spike。
- `W1-05` 头条号发布页填充 Spike。

本周验收：

- 本地开发模式前后端可运行。
- 数据库迁移可执行。
- 可保存并复用一个头条号登录状态。
- 有头条号发布页 URL、标题/正文/封面填充策略和失败点记录。

## Week 2：文章、资源、分组、账号

目标是让内容生产和账号授权真正可用，为 Week 3 的发布任务提供数据。

本周不可延期：

- 文章 CRUD。
- 资源上传与访问。
- TipTap JSON/HTML/plain_text 保存。
- 正文图片顺序同步。
- 分组 CRUD 与分组成员维护。
- 账号授权、校验、重新登录。

本周验收：

- 能完整创建、编辑、删除文章。
- 能上传封面和正文图片，刷新后仍可查看。
- 能创建和维护文章分组。
- 能添加并校验头条号账号。
- 授权包导出有最小可用结果，或明确剩余工作进入 Week 4。

## Week 3：任务、日志、发布主链路

目标是打通 MVP 核心价值链路：文章 -> 账号 -> 任务 -> 半自动填充 -> 人工发布确认 -> 结果记录。

本周不可延期：

- 单篇任务。
- 分组轮询任务。
- 发布记录生成。
- 任务执行器和日志。
- `ToutiaoPublisher` 集成。
- 等待人工发布、手动确认、失败继续、失败重试。

本周验收：

- 单篇任务能自动填充头条号发布页并停在发布前。
- 分组任务能按文章顺序轮询多个账号。
- 任务日志能在前端查看。
- 用户能手动标记成功/失败。
- 失败记录能重试。

## Week 4：打包、验收、修复

目标是交付可给产品同事试用的 Windows MVP，而不是继续扩功能。

本周不可延期：

- React build 与 FastAPI 静态托管整合。
- exe 打包。
- Playwright Chromium 分发。
- 干净 Windows 环境验证。
- 主链路手工验收。
- 交付说明和已知问题。

本周验收：

- 双击 exe 可启动本地服务并打开浏览器。
- 数据保存到本地数据目录。
- 头条号浏览器状态可保存和复用。
- 完整主链路可跑通。
- 授权包可导出。
- 有基础使用说明和已知问题列表。

## 时间缓冲策略

如果 Week 1 的头条号 Spike 失败或不稳定，Week 2 仍继续做文章、资源、账号基础能力，同时将发布自动化改为更强人工兜底。

如果 Week 3 结束时头条号自动填充仍不稳定，应砍掉以下项保证核心交付：

- Markdown 导入。
- Word 图片自动粘贴。
- 系统状态高级信息。
- 任务日志高级筛选。
- 分组排序 UI。

不可砍：

- 文章保存。
- 封面上传。
- 账号授权保存。
- 单篇发布填充或人工填充兜底。
- 发布记录。
- Windows 一键启动。
