# Session 记录：Geo 平台 Web 化架构设计

**日期**: 2026-05-11  
**主题**: 从 Windows exe 转向纯服务器 Web 版本的技术方案

## 背景

- 项目现状：Windows 桌面 MVP（FastAPI + React + Playwright），打包为 `.exe`
- Leader 需求：改成网页形式（不接受 exe）
- 决策：坚持纯服务器部署方案（非混合方案）

## 核心挑战讨论

### 1. 初期考虑的方案及问题

#### 方案 A: 纯服务器 + 无头浏览器
- **问题**:
  - 头条号等平台有反爬虫检测，容易拒绝无头浏览器
  - 用户账号认证困难（涉及验证码、二次认证）
  - 账号被封禁/退出后难以重新登录
  - 无法处理动态验证（如扫码）

#### 方案 B: 保留本地 exe，改成 Web 界面访问
- **优势**: 解决了无头浏览器的所有问题
- **劣势**: 不符合 leader 的纯服务器需求

#### 方案 C: 采用公司已采购的同类软件
- **发现**: 公司之前采购过类似功能的软件，采用"网页登录 + 本地 exe 自动化"的混合模式
- **结果**: Leader 和 PM 仍坚持自主开发纯服务器方案

### 2. 关键转折：交互式需求

用户认识到项目**不能完全自动化**，需要支持：
- 账号重新登录（需要用户操作）
- 二维码扫码（必须人工操作）
- 关闭动态弹窗（逻辑复杂，自动化脆弱）

**结论**: 必须采用**有头浏览器**，支持实时人工交互

## 最终架构方案

### 整体设计

**交互式远程浏览器**：用户在 Web 界面中实时操作服务器上的有头 Chrome 浏览器

```
Web 前端 ←→ 服务器（Chrome + Xvfb）

流程：
1. 用户创建任务 → 服务器启动 Chrome 实例
2. 自动化运行 → 遇到"暂停点"（如登录、扫码、弹窗）
3. 前端实时显示浏览器画面（直播流）
4. 用户在 Web 界面交互（扫码、点击、输入）
5. 操作转发到 Chrome → 继续自动化
6. 完成/失败
```

### 技术栈

| 组件 | 方案 |
|------|------|
| 操作系统 | Linux（Ubuntu 20.04+） |
| 虚拟显示 | Xvfb（X Virtual FrameBuffer） |
| 浏览器 | Chromium + Playwright |
| 画面转发 | WebSocket + 定时截图（简单）或 noVNC（实时） |
| 操作转发 | Web 坐标 → Playwright 鼠标操作 |
| 任务管理 | 异步队列（Celery/RQ 或 threading） |
| 浏览器池 | 支持多并发实例管理 |

### 数据流

```
用户操作
    ↓
Web 前端 API: POST /api/tasks/{id}/browser-action
    ↓
后端转发坐标 → Playwright 执行鼠标/键盘操作
    ↓
Chrome 响应 → 截图
    ↓
后端检查"暂停点" → 决定是否继续或等待用户
    ↓
WebSocket 推送画面到前端
```

### 核心改动点

#### 后端
1. 改任务执行为异步队列（目前是同步）
2. 加入浏览器实例池管理
3. 新增暂停点检测机制（登录页、扫码、特定弹窗）
4. 新增 API: `POST /api/tasks/{id}/browser-action`（用户交互转发）
5. 新增 API: `GET /api/tasks/{id}/screenshot`（实时截图）
6. 任务状态扩展：`in_progress` → `waiting_user_input` → `in_progress`

#### 前端
1. 实时显示浏览器画面（WebSocket 或轮询）
2. 鼠标点击映射到服务器坐标
3. 输入框/二维码识别界面
4. 任务进度和状态展示

#### 部署
1. Docker 容器：Xvfb + Chromium + Python 运行环境
2. 支持多容器/多进程支持并发任务
3. 容器编排（Docker Compose 或 Kubernetes）

### 资源估算

| 并发数 | 内存 | CPU | 说明 |
|--------|------|-----|------|
| 1 | ~1GB | 1核 | Chrome + Xvfb 开销 |
| 5 | ~5GB | 4核 | 5 个浏览器实例 |
| 10+ | 需要扩展 | 扩展 | 考虑负载均衡 |

## 下一步

1. **后端**: 实现异步任务队列、暂停点检测、浏览器实例管理
2. **前端**: 实现远程浏览器画面显示和交互
3. **部署**: 准备 Docker 配置、Xvfb 设置
4. **测试**: 在头条号上验证有头浏览器是否能通过反爬虫检测

## 注意事项

- 有头浏览器方案**资源成本较高**，需要合理规划服务器配置
- **运维复杂度增加**，需要监控浏览器实例、内存泄漏、僵尸进程
- 用户交互的实时性取决于网络延迟和截图更新频率
- 需要设计合理的**超时机制**（防止用户长时间不操作导致资源占用）
